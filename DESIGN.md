# Design: Meditation Timer (personal, offline PWA)

Generated by /office-hours on 2026-07-05
Repo: Meditation-Timer (greenfield, no code yet)
Status: APPROVED
Mode: Builder

## Problem Statement
A meditation timer built for one user (you). You sit down, tap start, and get a
calm, reliable timer with a lovely bell. It must work on your phone, fully
offline, with no internet connection required during a sit. Simplicity is the
explicit goal.

**Motivation:** the prior timer became part of a larger platform that added
scope beyond timing. This project's guiding principle is to stay a single-purpose
timer. See Non-Goals for the explicit boundaries.

## What Makes This Cool
The bell is the product. Not a countdown with a beep bolted on — a genuinely
pleasant, reliable bell (recorded singing bowl) framed by a screen calm enough
to settle you before the first strike. Local-only, no accounts, works on the
home screen even in airplane mode. The appeal is focus: it does one thing well
and stays that way.

## Constraints
- **Phone-first.** Used on a phone, likely sitting on the floor or nearby.
- **Offline-mandatory.** No network during a session. Everything cached locally.
- **No App Store.** Browser-based PWA, added to home screen.
- **Simple.** One user, one purpose. Resist feature creep.

## Non-Goals (out of scope by design)
These are intentional boundaries, recorded so scope stays stable over time. Not
"never possible," but explicitly not part of this project's direction:
- User accounts, login, or cloud sync.
- Push notifications or reminders.
- Engagement mechanics: streaks-as-pressure, badges, points, nudges.
- Surveys, ratings prompts, or feedback capture.
- Analytics, telemetry, or any network calls during use.
- Social/sharing features.

Deferred but not out of scope (candidate for v2/v3): interval bells (v1 optional),
ambient soundscapes, simple private session history.

## Premises (all agreed)
1. **v1 is a bell timer, nothing else.** Pre-roll delay, duration, start bell,
   optional interval bells, end bell, calm countdown screen, plus pause/resume
   and end-early controls. Ambient soundscapes and streak tracking are
   deliberately deferred to v2/v3 (see Non-Goals for what is out of scope
   entirely).
2. **PWA is the vehicle.** Service worker caches everything locally; installs to
   home screen; works fully offline after first load.
3. **Hosting up front.** Serve once over HTTPS on a free static host (GitHub
   Pages / Netlify / Vercel), load on phone once, cached forever.
4. **Keep-awake + audio reliability are core, not afterthoughts.** Screen must
   stay on during a sit; the end bell must always fire.

## Approaches Considered
### Approach A: Single-file vanilla PWA + recorded bell (CHOSEN)
One `index.html` (inline CSS + JS), a `manifest.webmanifest`, a service worker,
and 1–2 high-quality recorded bell samples. No framework, no build step.
- Effort: S | Risk: Low
- Pros: ships in a day; recorded bell = guaranteed lovely sound; trivial to host
  and cache offline; edit one file and refresh.
- Cons: single file gets crowded once ambient + tracking arrive; must source
  1–2 CC0/CC audio files.

### Approach B: Vite + Svelte PWA scaffold
Proper project with vite-plugin-pwa, components, TypeScript, built to grow into
ambient audio + tracking.
- Effort: M | Risk: Med
- Deferred: correct destination only once v2/v3 features actually arrive and the
  single file starts to hurt. Overkill for v1.

### Approach C: Zero-asset synth bell (Web Audio)
Same single-file app, but the bell is generated in code — no audio files at all.
- Effort: S–M | Risk: Med (sound quality is a gamble)
- Deferred: a fun later experiment; the loveliness bar makes a recorded sample
  the safer day-one choice.

## Recommended Approach
**Approach A.** The bell is the whole point, so a recorded singing-bowl sample
guarantees it sounds beautiful on day one with no gamble. Single-file vanilla
keeps building, hosting, and hacking dead simple — matching the "simplicity"
goal. Graduate to B's structure only if/when tracking + ambient audio make the
single file painful.

## Technical Design Notes (v1)
- **Structure:** `index.html`, `manifest.webmanifest`, `sw.js` (service worker),
  `/audio/bowl.mp3` (+ optional second bell). Optional `icons/` for home screen.
- **Controls:** Start, Pause/Resume (toggle), End (early). Keep the control set
  tiny and unmistakable — big touch targets, calm styling.
- **Pre-roll delay:** a settable 5–10s countdown (default ~7s) *before* the
  start bell, so you can get situated. The start bell marks the true beginning
  of the sit; the session duration counts from there, not from the delay.
- **Timer core:** compute an absolute end timestamp (`Date.now() + remaining`);
  render remaining time off that, not off a decrementing counter. Survives tab
  throttling and clock drift far better than `setInterval` counting.
- **Pause/Resume:** on pause, store the remaining seconds and clear scheduled
  bells; on resume, recompute a fresh end timestamp (`Date.now() + remaining`)
  and reschedule any interval/end bells. This keeps drift-resistance while
  supporting real interruptions.
- **End early:** stop the timer, cancel scheduled bells. Decide once: does
  ending early ring the end bell (a gentle close) or exit silently? Recommend
  ringing it — a clean bell to close the sit feels better than an abrupt stop.
- **Bells (implemented via Web Audio scheduling):** on the Start tap, create/
  resume an `AudioContext` and decode the bell into a buffer. Then schedule BOTH
  bells on the audio clock up front — start bell at `currentTime + preroll`, end
  bell at `currentTime + preroll + session`. The audio thread fires scheduled
  sources on time even when the page isn't rendering, and the Start tap's audio
  permission carries forward to bells that ring seconds later. This is what makes
  a delayed bell actually sound. (Original plan was to trigger bells from a JS
  timer — that failed: the timer stalls when the page isn't visible, so the bell
  never fired. Audio-clock scheduling is the fix.)
- **Timer/display vs. bells are decoupled:** the MM:SS countdown runs on
  `setInterval` (keeps firing, throttled, in the background — unlike
  requestAnimationFrame, which stops dead). If the display stalls while hidden,
  the bells still ring; when the page becomes visible again the display catches
  up from the absolute end timestamp.
- **Pause/resume:** on pause, cancel the scheduled end-bell source and store
  remaining ms; on resume, reschedule the end bell at `currentTime + remaining`
  and recompute the display end timestamp.
- **Screen stays on:** Wake Lock API (`navigator.wakeLock.request('screen')`),
  re-acquired on `visibilitychange`. Keeps the display alive and the audio
  context running; it's a comfort layer now, not the sole bell guarantee.
- **Offline:** service worker caches the HTML, manifest, JS, and audio on
  install; serve cache-first so a sit never touches the network.
- **State:** duration + interval settings in `localStorage`. No backend.

## Success Criteria
- You can add it to your phone's home screen and open it with no network.
- Set a duration + pre-roll, tap start, get situated during the delay, hear a
  lovely start bell.
- Pause/resume survives an interruption without losing the sit; end-early works.
- Screen stays on for the whole sit.
- End bell reliably fires at the end, every time.
- Scope holds: the Non-Goals list still describes the app six months from now.
- You reach for it tomorrow morning instead of your old timer.

## Distribution Plan
- **Host:** push repo to GitHub, enable GitHub Pages (free, HTTPS). One-time
  load on your phone caches it for offline use.
- **Install:** Safari/Chrome → Share → "Add to Home Screen."
- **CI/CD:** optional GitHub Action to deploy Pages on push; for a single-file
  app, even manual "commit to main" is fine to start.

## Open Questions

  Pixabay, or record your own bowl if you have one.)
- Default pre-roll length? (5, 7, or 10s — and fixed vs. adjustable.)
- Do you want a warning/interval bell in v1, or truly just start + end?
- Default duration(s)? (e.g. 10 / 20 min presets vs. a free dial.)
- End-early: ring the end bell or exit silently? (Recommend ring.)

## Next Steps (build order)
1. **Scaffold the single file.** `index.html` with a start button, a duration
   input, and a big calm countdown. Absolute-timestamp timer. No bell yet — get
   the count right first.
2. **Add the controls.** Pre-roll delay before the start, plus Pause/Resume and
   End. Verify pause stores remaining time and resume recomputes cleanly.
3. **Add the bell.** Drop in one recorded bowl sample, unlock audio on the Start
   tap, fire start + end bells against the timeline. This is the "whoa, it
   works" moment.
4. **Keep the screen awake.** Wire up Wake Lock; test a full 10-minute sit on
   your actual phone without the screen dimming.
5. **Make it a PWA.** Add `manifest.webmanifest` + icons + a cache-first service
   worker. Verify "Add to Home Screen" and a full offline sit (airplane mode).
6. **Host it.** GitHub Pages, load once on your phone, done.
7. *(Later, only if you want it)* interval bells → ambient audio → streak
   tracking. That's when you'd consider refactoring toward Approach B.

## The Assignment
Before writing any code: **source the bell.** The whole product hinges on one
sound you'll be happy to hear every morning. Go find 2–3 candidate singing-bowl
samples (Freesound/Pixabay, CC0), or record your own bowl if you have one, and
actually listen to them on your phone speaker at meditation volume. Pick the one
that makes you exhale. Drop the file in `/audio/` and note the choice here.
Everything else is plumbing around that sound.

  straight to "the bells." You cut your own feature list down faster than most
